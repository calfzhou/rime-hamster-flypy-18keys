## Package layout

Current skin root:

- `skin/hamster-18key-minimal/`
  - `config.yaml`
  - `demo.png`
  - `light/`
  - `dark/`

`light/` and `dark/` have the same file set (1:1), only style/theme values differ.

## File types

### 1) `*.yaml` (editable source of skin behavior)

These are the files we should modify.

Common categories:

- `pinyin_*` â†’ Chinese keyboard layout/behavior
- `alphabetic_*` â†’ English keyboard
- `numeric_*` â†’ Number keyboard
- `symbolic_*` â†’ Symbol keyboard
- `hamstersetting` â†’ floating panel

### 2) `*.keyboard` (generated artifacts, not tracked)

These are dotfiles like:

- `.pinyin_26_portrait.keyboard`
- `.alphabetic_26_portrait.keyboard`

Per the author note, they are auto-generated by Hamster and normally **do not need manual edits**.

Observed in this package:

- `.keyboard` content is plain text, but appears to be expanded/processed form of `.yaml`
  (larger line counts, anchors/comments mostly flattened).

Current repo policy:

- `.keyboard` files are treated as disposable cache files.
- They are **not required** for packaged `.hskin` to work.
- They are **not kept in git**.

Recommendation: edit `.yaml` only.

### 3) `resources/*` (optional theme graphics)

This skin version no longer uses the old floatsearch panel assets.

## Entry point: `config.yaml`

`config.yaml` maps keyboard types to concrete files by device/orientation (iPhone/iPad, portrait/landscape/floating).

When adding a new keyboard variant (e.g. flypy-specific 18-key page), this mapping is the first place to wire it in.

## Key editable fields inside keyboard YAML

Typical high-impact sections:

- `rimeSchema:` binds keyboard to schema id
- `preeditHeight`, `toolbarHeight`, `keyboardHeight`
- `toolbar:` button actions
- `keyboard:` row/cell layout
- `*Button` blocks (`action`, `swipeUpAction`, `swipeDownAction`, `holdSymbolsStyle`)

## Theme update rule (important)

Because `light/` and `dark/` are parallel and both active in real use:

1. Structural/functional changes should be mirrored in **both** themes.
2. If only color/visual values differ, keep behavior keys aligned.
3. After each change, verify filenames and structure still match between `light/` and `dark/` (for source files, primarily `.yaml`/assets).

Practical check command:

```bash
cd skin/hamster-18key-minimal
find dark -type f ! -name '*.keyboard' | sed 's#^dark/##' | sort > /tmp/dark_files.txt
find light -type f ! -name '*.keyboard' | sed 's#^light/##' | sort > /tmp/light_files.txt
comm -3 /tmp/dark_files.txt /tmp/light_files.txt
```

No output means file inventories are still symmetric.

## Recommended modification workflow

1. Edit target `.yaml` in `light/`.
2. Apply equivalent structural change in `dark/`.
3. Ignore `.keyboard` files; focus on `.yaml` + assets.
4. Build package with:
   - `make hskin`
5. Test in Hamster.
